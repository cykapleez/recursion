# Godot Client Log — 2026-05-07

### Session — Camera Zoom (Mouse Scroll Wheel)

**Asked:** Allow the player to zoom via the mouse scroll wheel. At least 3 zoom levels in, 3 levels out.

**Done:**
- Added 7-level zoom constant array (`ZOOM_LEVELS`) and `ZOOM_DEFAULT` index (3) to `player.gd`
- Added `CAMERA_BASE` constant matching the camera's rest position in `Player.tscn` (`Vector3(0, 12, 9)`)
- Added `_zoom_index` and `_camera` instance variables
- Added `call_deferred("_find_camera")` in `_ready()` to grab the Camera3D child after scene is ready
- Added `_find_camera()` and `_apply_zoom()` functions — zoom applies with a 0.12s EASE_OUT tween
- Added scroll wheel handling at the top of `_unhandled_input()` before the dead/charging guard, so zoom works in all modes
- Fixed `_screen_shake()` in `enemy.gd`: changed hardcoded `Vector3(0, 12, 9)` rest position to `cam.position` so the shake returns to the current zoom level instead of always snapping back to default

**Files changed:**
- `scripts/player.gd` — added zoom constants, variables, `_find_camera()`, `_apply_zoom()`, scroll wheel input
- `scripts/enemy.gd` — fixed `_screen_shake()` rest position to respect current zoom level

**Decisions:**
- Zoom levels use a multiplier array applied to `CAMERA_BASE` rather than absolute positions, so a single constant defines the anchor and all 7 levels scale from it
- Scroll wheel is handled before the `_dead` guard so players can zoom even after respawning during the brief visible=false phase — minor but intentional
- `MOUSE_BUTTON_WHEEL_UP` decrements the index (zooms in / moves camera closer) which matches industry-standard scroll-to-zoom convention

**Notes:**
- Camera node is expected as a direct child named `Camera3D` on the Player node. If the scene hierarchy changes, `_find_camera()` needs updating.
- Screen shake in enemy.gd now correctly returns to whatever zoom the player is at, not the default. Previously any AOE slam would jolt the camera back to index 3 regardless of zoom setting.

### Session — Player Running Animations

**Asked:** Player character has animation GLBs (running, walking, throw, etc.) in the builder folder. Align/wire the animations to the player character.

**Done:**
- Replaced the texture-only model (`0507225504_texture.glb`) with the rigged running model (`Running_withSkin.glb`) which has a Skeleton3D + AnimationPlayer with the run animation embedded
- Added `_setup_animations()`: discovers the run animation name at runtime from the default library, then loads `Walking_withSkin.glb` temporarily, deep-copies its AnimationLibrary into the base model's AnimationPlayer under the name `"walk"`, then frees the temp instance
- Added `_update_animation(delta)`: rotates `_model` to face movement direction using `lerp_angle` + `atan2(-dir.x, -dir.z)`; plays walk at low speed (<3 u/s), run at full speed, stops when idle
- Wired `_update_animation(delta)` into both branches of `_physics_process()` (normal movement and smash lunge)
- Added `_model`, `_anim_player`, `_anim_run`, `_anim_walk` variables

**Files changed:**
- `scripts/player.gd` — replaced model GLB, added animation constants, `_setup_animations()`, `_update_animation(delta)`, new vars

**Decisions:**
- Used `Running_withSkin.glb` as the base model rather than `biped_Character_output.glb` because it is guaranteed to have the rig + AnimationPlayer + running animation in one file, with no guessing about whether the base output has an AnimationPlayer
- Animation names are discovered at runtime (`names[0]` from the default library) rather than hardcoded, because Meshy AI export names vary (e.g., "Armature|Running" vs "Running" vs "mixamo.com")
- Walking animation is stored in a named library `"walk"` so it coexists with the default library from the running GLB; playback uses `"walk/animname"` format
- Model rotation uses the same `atan2(-dir.x, -dir.z)` formula as enemy.gd to keep conventions consistent

**Notes:**
- If the walking animation doesn't play (walks in T-pose), the Walking GLB's skeleton node hierarchy doesn't match the Running GLB's. This is a Meshy export artifact — all tracks in the walking animation reference paths relative to that scene's root. The scene structures must be identical for cross-GLB animation copying to work. If this happens, the fallback is to import both animations into a single GLB in Godot's editor AnimationLibrary importer.
- The available animation GLBs not yet wired: `RunFast_withSkin`, `Run_02_withSkin`, `Female_Crouch_Pick_Throw_Forward_withSkin`, `Over_Shoulder_Throw_withSkin`. These can be added later for sprint and throw animations.
