# Godot Client Log — 2026-05-05

### — Build preview ghost

**Asked:** When selecting an item to build in the build menu, show a transparent version of the item where the mouse cursor is.

**Done:**
- Created `scripts/build_preview.gd` — Node3D that instantiates a WorldObject ghost, strips collision and labels, makes all materials 40% transparent, and follows the snapped (or ALT=free) mouse position in real time.
- Updated `scenes/World.tscn` — bumped `load_steps` from 13 to 14, added ext_resource id="10" for build_preview.gd, added `BuildPreview` Node3D as a sibling of ThrowArc3D.

**Files changed:**
- `scripts/build_preview.gd` — new file; connects to `GameManager.mode_changed`, caches player ref, rebuilds ghost when `build_object_id` changes, clears ghost when leaving BUILD mode.
- `scenes/World.tscn` — added ext_resource and BuildPreview node.

**Decisions:**
- Ghost is built by instantiating WorldObject.tscn and calling `setup()` on it — reuses all existing visual logic (box/tree/house) without duplicating it.
- After setup: `remove_from_group("world_objects")` (not pickupable), `collision_layer/mask = 0` (no physics), all Label3D hidden, all MeshInstance3D materials duplicated with `TRANSPARENCY_ALPHA` and alpha=0.4.
- Rebuilds ghost only when `build_object_id` changes, not every frame — avoids per-frame instantiation.
- Used `Player.Mode.BUILD` constant directly (works because player.gd has `class_name Player`).

**Notes:**
- `_player.get("mode")` returns Variant; typed as `int` explicitly — GDScript 4 unboxes at runtime.
- `_make_transparent` recurses into child Node3Ds, handling nested meshes (e.g. house roof panels, tree trunk+canopy).
- Ghost position snaps to grid unless KEY_ALT is held — mirrors the placement behaviour in player.gd.

### — Thrown object visuals match original model

**Asked:** When objects are thrown, they should look exactly like the object that was picked up. Hitbox can stay small.

**Done:**
- Replaced `_build_mesh()` in `throw_projectile.gd` with `_build_visual()` — instantiates `WorldObject.tscn`, calls `setup(object_id, object_def)`, then reparents only the visual children onto the `RigidBody3D`. Collision shapes and labels are skipped.
- The `ThrowProjectile.tscn` `BoxShape3D` (0.5 × 0.35 × 0.5) is unchanged — stays as the physics hitbox per the user's request.

**Files changed:**
- `scripts/throw_projectile.gd` — `_build_mesh()` removed; replaced by `_build_visual()`. The BoxMesh + flat-color material is gone. The object now renders as the same GLB (or box fallback) at the same scale as the placed version.

**Decisions:**
- Used `wo.get_children()` snapshot before the loop to avoid mutation-while-iterating issues.
- Only reparent children that are not `CollisionShape3D` or `Label3D` — keeps the StaticBody3D's physics out of the RigidBody3D's physics tree and removes floating label nodes during flight.
- `wo.queue_free()` after stealing children — the empty WorldObject shell is discarded immediately.

**Notes:**
- The GLB model's visual exactly matches the world object: same scale (including `_SCALES` overrides for shed/house), same materials, same textures. What you see in the sandbox is what flies.
- Tumble spin is still weight-scaled (heavy = slower spin). This now looks physically correct with the real model tumbling in 3D.

### — Texture system upgrade (PBR pass)

**Asked:** Audit textures, define art style, create an asset manifest, implement modular texture loading, replace placeholders with existing files, and report TODOs for missing assets.

**Done:**
- Audited all visual nodes — player/enemy are fully procedural; world objects use Kenney GLBs; ground is a flat-color PlaneMesh.
- Added `roughness` and `metallic` optional parameters to `_mat()` in both `player.gd` and `enemy.gd` (defaults preserve old behaviour at call sites that weren't updated).
- Updated `player.gd` `_build_visuals()` material calls with physically tuned values: cloth (roughness 0.85), skin (0.70), eye gloss (0.35, metallic 0.15).
- Updated `enemy.gd` `_build_leg()` and `_build_visuals()` material calls: decayed flesh (roughness 0.95), yellowed bone (0.78), dark iron armor (0.32, metallic 0.80), oxidised spine (0.45, metallic 0.55).
- Updated `scenes/World.tscn` ground `StandardMaterial3D` — added `roughness = 0.95`, `metallic = 0.0`, `metallic_specular = 0.1` to reduce the plastic sheen.
- Added TODO comments in player.gd and enemy.gd flagging where character texture sheets should be applied when sourced.

**Files changed:**
- `scripts/player.gd` — `_mat()` now accepts `roughness` and `metallic`; `_build_visuals()` call sites updated.
- `scripts/enemy.gd` — same signature upgrade; `_build_leg()` and `_build_visuals()` updated with per-material physical values; armour calls now use metallic 0.80.
- `scenes/World.tscn` — ground material has roughness/metallic properties; load_steps unchanged.

**Decisions:**
- Did not apply `colormap.png` to the ground. The Kenney colormap is a UV palette atlas — not a seamless tileable texture. Applying it would produce a patchwork of colored squares across the 80×80 ground plane. Held off until a proper grass/dirt texture is sourced.
- Did not create a TextureManager autoload — with only procedural characters and Kenney GLBs (textures embedded), there is no runtime texture lookup yet. The autoload would be empty. Flag for when character texture sheets are added.
- GLB textures (colormap.png, texture-a through texture-r) are already imported by Godot and referenced inside the GLB files. No extra wiring needed in world_object.gd.

- `build_preview.gd` uses `mat.duplicate()` — PBR properties carry through to ghost materials automatically. No change needed there.
